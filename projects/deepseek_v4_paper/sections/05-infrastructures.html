Reading Angle: Co-Design as the Unifying Thread

This section reads like a catalog of infrastructure components, but the deeper logic connecting them is co-design — every subsystem is shaped by a specific hardware constraint, and every optimization is a negotiation between performance and numerical fidelity. The authors are not simply throwing faster kernels at the problem; they are rethinking the boundary between what the algorithm demands and what the hardware can deliver.

Consider the through-line: EP wave scheduling exists because computation time exceeds communication time in MoE layers, making overlap theoretically possible — but only if you slice experts finely enough to keep the pipeline full. TileLang exists because hand-writing hundreds of fused CUDA kernels is unsustainable, yet off-the-shelf ATen operators leave too much performance on the table — the Z3 integration and host codegen are specifically there to close the gap between DSL productivity and hand-tuned quality. Batch invariance and determinism exist because non-deterministic training makes debugging impossible at scale, and the cost of atomicAdd-based accumulation is paid in debugging time, not FLOPs. The Muon-ZeRO hybrid exists because Muon's matrix-level updates are fundamentally incompatible with element-wise parameter partitioning — a classic algorithm-system mismatch.

The KV cache design is perhaps the most emblematic: hybrid attention breaks PagedAttention's assumptions, so the authors treat SWA + uncompressed tails as a state-space model — a conceptual reframing that unlocks a fixed-allocation strategy. This is co-design in its purest form: changing the mental model of what the cache is to make it manageable.

The computation-communication ratio insight — each GBps of bandwidth hides 6.1 TFLOP/s of compute, and beyond that threshold, more bandwidth yields diminishing returns — is a design principle for future hardware, not just a tuning knob for current systems.

Why the Muon–ZeRO conflict matters

Standard ZeRO partitions a parameter matrix element-wise across ranks — each rank owns a slice and updates it independently. Muon, however, needs the full gradient matrix to compute the orthogonalization via Newton-Schulz iterations. You can't orthogonalize a matrix you only see a slice of. The authors' knapsack-based bucket assignment is a pragmatic escape: keep entire matrices on each rank (limiting ZeRO parallelism for dense params), pad to equal sizes for efficient reduce-scatter, and accept ~10% memory overhead. For MoE params, the sheer number of experts makes per-matrix partitioning natural — flatten all down-projections, then up-projections, then gates, and distribute the flat vector without splitting any logical matrix. The BF16 gradient quantization with stochastic rounding + all-to-all-then-local-FP32-sum is a clever way to halve communication without sacrificing numerical robustness.
